# Claude instructions

## Project overview
PhyloHPC is a full phylogenomics pipeline that goes from multi-species proteomes to homology groups, gene trees, ortholog calls, species-level annotation tables, and interactive HTML reports.

## Manual summary
- `docs/manual.md` is still useful for the intended repository contract:
  - `step1.nf` for PFAM/domain search plus HG clustering
  - `step2.nf` for alignment, gene trees, POSSVM, and optional GeneRax
  - `workflow/gather_annotations.py` for per-species ortholog annotation tables
  - local and SLURM execution as first-class modes
  - `ids.txt` as the main handoff between step 1 and step 2
  - optional `resources.tsv` resource prediction for large step-2 runs
- Treat the manual as high-level intent, not precise implementation detail. Prefer the current code when behavior differs.

## Stream selection
- Before doing substantial work, ask the user which stream to work on unless they already made it explicit.
- The stream names are:
  - `pipeline`
  - `report`
  - `hogs`
- If the user does not specify a stream but the request clearly maps to one of them, say which stream you are assuming and proceed.
- Once a stream is chosen, restrict attention to that stream's files and workflows unless the user explicitly asks for cross-stream work or the bug clearly crosses the boundary.
- Read the corresponding stream file and treat it as the main directive:
  - `instructions/pipeline.md`
  - `instructions/report.md`
  - `instructions/hogs.md`
- For report work, `instructions/report.md` should also track current UI defaults, regeneration expectations, and Python compatibility constraints of `make_report.sh`, not just broad architecture notes.
- If a change meaningfully updates intended functionality and the user approves that behavior, add or refine the relevant instruction in the stream file so another agent can follow it unambiguously.

## Streams
- `pipeline`
  - scope: improve the main workflow from proteomes through HG selection, trees, orthology, annotations, and step-2 reporting
- `report`
  - scope: improve the interactive step-2 HTML report generated by `workflow/report_step2.py`, including the family-centric domain-architecture browser and the per-protein domain popups fed by `results/search/*.domains.csv` and `*.domains_ummerged.csv`
- `hogs`
  - scope: improve and stabilize step 4 hierarchical orthogroup inference and its HTML report

## Shared repo rules
- Preserve the recognizable user workflow: install env, run step 1, filter HGs, run step 2, gather annotations, then run or test step 4 reporting as needed.
- `ids.txt` is the main selected-HG handoff into step 2. `ancestry_ids.txt` is the focused subset for step 4 and hOG reporting.
- `resources.tsv` is an optional step-2 optimization input, not part of the biological contract.
- If both `results/possvm/` and `results/possvm_prev/` exist, treat OG names, gene-to-OG assignments, reference orthologs, OG support, and reference support as source-specific unless a caller explicitly wants a merged summary.
- `workflow/gather_annotations.py` is part of the intended pipeline functionality and should keep working against current POSSVM and search outputs.
- When docs and code disagree, prefer the active code for the chosen stream.
